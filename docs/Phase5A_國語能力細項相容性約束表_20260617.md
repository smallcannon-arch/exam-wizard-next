## 〇、定位與邊界

本表只回答：**某能力細項，可以用哪些「承載形式」與「作答形式」被合理評量？**
它**不**決定「整份卷怎麼湊到 100 分／向度比例怎麼成立」——那是主流程的責任：

```
abilityBudget / targetScore（向度比例驅動）
  → slot allocator（依預算配題位）
  → compatibility check（用本約束表）
  → teacher override / warning
  → AI content generation（只生內容）
```

相容是**必要非充分**條件；向度比例仍由 allocator 依 targetScore 配置並由題位統計驗證。

### 配題優先序（防止「題型多元」反壓能力預算）

```
1. 先滿足 targetScore / abilityBudget。
2. 再檢查 ability 與 carrier / response 是否相容。
3. 再處理教師的題型偏好。
4. 最後才追求題型多元與難易梯度。
衝突規則：
- 相容性 vs 題型多元 → 以相容性為先。
- 相容性 vs targetScore → 系統示警、請教師調整，不由 AI 自行補救。
```

## 一、三層概念：carrier / response / 旗標

| 層級 | 欄位 | 例子 | 意義 |
|------|------|------|------|
| 承載形式 | `carrierType` | standalone、reading_set、competency_set | 是否需共用文本／情境（題組） |
| 作答形式 | `responseType` | multiple_choice、fill_blank、phonetic、char_writing、error_correction、reordering、sentence_modeling、short_answer、composition | 學生怎麼作答 |
| 旗標（推導，不存細項） | `requiresSharedMaterial`／`requiresRubric` | — | 由 carrier／response 的 meta 表推導 |

- `requiresSharedMaterial` 由 **carrier** 決定（見 CARRIER_TYPE_META），**不**存進每個細項。
- `requiresRubric` 由 **response** 決定（見 RESPONSE_TYPE_META），**不**放進政策欄。
- 〔待釐清〕「圖表判讀題」現行為「選擇題形式、表格嵌題幹」的單題 → `responseType: multiple_choice` + `chartEmbedded` 旗標、carrier 仍 standalone。**邊界**：若同一圖表供多個子題共同作答，才升格為 carrier（如 `chart_set`）；單題圖表仍以 standalone + chartEmbedded 處理。
> 注意：`chartEmbedded`／`chart_set` **僅止於文件備註，Phase 5A 不納入資料表**，以免把國語相容表擴成跨科圖表題模型、範圍飄移。

### 鍵值穩定性與相容模型

- `CHINESE_ABILITY_COMPATIBILITY` 的 key 暫以 `CHINESE_AUDIT_STRUCTURE` 中唯一且穩定的中文細項 label 作為 `abilityId`；一經進入題位資料即視為**穩定資料鍵**，不得因 UI 文案潤飾任意變更；未來細項改名須提供 migration map。
- Phase 5A 採 `allowedCarriers × allowedResponseTypes` 的**寬鬆相容模型**（笛卡兒積皆視為相容）。若校核發現某細項並非所有組合皆合理，後續可升級為 `allowedPairs` 精準模型（先不做，留路）：

```js
allowedPairs: [
  { carrier: "standalone",  response: "multiple_choice" },
  { carrier: "reading_set", response: "multiple_choice" }
]
```

### selectionPolicy（取代含糊的 locked）

| enum | 意義 | 誰能改 |
|------|------|--------|
| `fixed_default` | 系統固定用 default | 老師要改需顯示警示 |
| `teacher_override_within_allowed` | 老師可在 allowed 內改 | 老師 |
| `allocator_select_within_allowed` | allocator 依整卷平衡在 allowed 內選 | 系統（**非 AI**） |

> `fixed_default` 覆寫細則：它代表系統預設固定用 default，**不等於完全不可改**。教師改為 allowed 集合內其他形式 → 黃色警示；改到 allowed 以外 → 紅色高風險，依 Phase 5C 規則處理。

第一階段 **AI 一律不選 carrier／response**，只生內容；不得改分數、形式、能力歸屬。

### 查無對照（絕不退回舊邏輯）

> ⚠️ 查無對照 → 標 `compatibilityUnknown`，**不**呼叫 `getChineseDimension`／`getChineseSubcategory`，要求教師手動指定 ability／dimension，列入高風險示警。底線：寧可停下問老師，不回到「題型推向度」。

### Helper 行為約定

- `getAbilityCompatibility(abilityId)`：查無時回傳 `null`，**不自行推導 dimension**。
- `isAbilitySlotCompatible({ abilityId, carrierType, responseType })`：相容性判斷的**權威入口**；查無 ability 時回傳 `{ ok: false, reason: "compatibilityUnknown" }`（或等價結構），交由 UI／稽核顯示高風險。
- `isCarrierCompatible` / `isResponseCompatible`：僅供單軸提示，**不得取代** `isAbilitySlotCompatible` 的整體 slot 檢查（未來由笛卡兒積升級為 `allowedPairs` 時，呼叫點不需更動）。

## 五、Phase 5 拆小：5A → 5D

- **5A**：只建相容表 + meta 表 + 測試（純資料）。
- **5B**：slot allocator 使用相容表；每 slot 的 `primaryAbility + carrierType + responseType` 必相容。
- **5C**：教師覆寫 + 稽核（見下表）。
- **5D**：AI 只生內容（carrier／response／ability 已定）。

> ⚠️ **Phase 5A 不會修掉截圖中的配分落差。** 那病根是「targetScore 是否真正驅動 slots 生成（含題型）」，屬 **5B**。5A 只守住「不出現句型辨識→國字題這種不相容配對」這道門檻。
> 開工序：① 封盤現有修補 → ② Phase 1／allocator 定位 → ③ 5A 純資料 + 測試 → ④ 5B 才讓 allocator 使用相容表。不要讓 Codex 誤以為「做完相容表配分就會準」。

### Phase 5C 教師覆寫規則

| 情況 | 系統反應 |
|------|----------|
| 改在 allowed 內 | 允許，重算 compatibility／score／dimension 稽核 |
| 改到 allowed 外 | 紅色高風險，不建議直接生成 |
| 教師堅持 | 要求改 primaryAbility，或填寫 override reason |
| 無 ability 對照 | `compatibilityUnknown`，不得靜默生成 |

## 六、Phase 5A 測試清單

```
1.  所有 CHINESE_AUDIT_STRUCTURE 細項都存在於 CHINESE_ABILITY_COMPATIBILITY。
2.  CHINESE_ABILITY_COMPATIBILITY 不得含 CHINESE_AUDIT_STRUCTURE 以外的細項（不可多）。
3.  細項 label 必須唯一；若不唯一，測試失敗並要求改 compound abilityId（如「段篇讀寫/閱讀理解/提取訊息」）。
4.  每筆 compatibility 的 dimension 必須與 CHINESE_AUDIT_STRUCTURE 攤平後該細項所屬 dimension 一致（不只是有填，還要填對）。
5.  allowedCarriers 不得為空。
6.  allowedResponseTypes 不得為空。
7.  defaultCarrier 必在 allowedCarriers 內。
8.  defaultResponseType 必在 allowedResponseTypes 內。
9.  selectionPolicy 必為合法 enum（fixed_default／teacher_override_within_allowed／allocator_select_within_allowed）。
10. Phase 5A 新增的 questionTypes.js 資料、compatibility helper 與其測試，不得呼叫或建立 getChineseDimension／getChineseSubcategory fallback；**亦不得新增以題型推導 ability／dimension 的替代 helper（換個名字做同樣的事）**。測試只檢查 Phase 5A 新增模組的 import 與 helper 行為，**不要求掃描整個舊專案**；舊有 getChineseDimension 是否存在，不作為 Phase 5A 測試失敗條件（移除留待 Phase 1／5B）。
11. 每個 allowedCarrier 必存在於 CARRIER_TYPE_META。
12. 每個 allowedResponseType 必存在於 RESPONSE_TYPE_META。
```

## 拍板結論
