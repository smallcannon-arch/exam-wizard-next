- 目的：讓 release owner 在 PR #1 / PR #2 stacked release train 與整合 PR 路線之間做明確決策。

## 2. Owner 要決定的問題

1. 是否要推進 release，而不是繼續維持 Draft。
2. 是否接受將 PR #1 + PR #2 合成一個 main 整合 PR。
3. 是否接受整合 PR diff 較大：34 files / 8441 insertions / 73 deletions。
4. 是否接受 raw output +180.1% 作為 cost / observability risk disclosure。
5. 是否知道 merge into main 仍會觸發 Pages deploy。
6. 是否已有 release risk review / rollback plan。
7. 是否要在 deploy 前先做同步生成進度 UI MVP。

## 3. 決策選項

### 選項 A：維持兩個 Draft PR

- 不建立整合 PR。
- 不 merge。
- 不 deploy。

優點：

- 最安全。
- 不會觸發 main deploy。
- 保留 PR #1 / PR #2 的 stacked review 脈絡。

缺點：

- release 不推進。
- 功能與穩定化修正繼續停留在 Draft PR。

適合：

- 尚未決定 deploy control。
- release risk 尚未被接受。
- owner 還不想進入 main release review。

### 選項 B：建立整合 PR 對 main

- 建立 `codex/release-prompt-quality-stabilized-20260621`。
- 分支來源為 `origin/codex/generation-stabilization-20260621` 的目前 head。
- base 為 `main`。
- PR 維持 Draft。
- 不 merge、不 deploy。

優點：

- 避免 PR #1 單獨進 main。
- reviewer 可直接看 PR #1 + PR #2 的最終狀態。
- release decision 可集中在一個 main-facing PR。
- 可在 PR body 內完整揭露 PR #1 / PR #2 的品質、效能與 raw output 風險。

缺點：

- diff 大，review 需要重新整理脈絡。
- PR #1 / PR #2 的歷史與驗收紀錄需在整合 PR body 補清楚。
- 即使建立整合 PR，也仍不能直接 deploy。

適合：

- owner 決定推進 release review，但不想讓 PR #1 單獨進 main。
- owner 尚未準備 deploy，但希望 main-facing review 開始整理。

### 選項 C：先控管 Pages deploy，再按 PR #1 -> PR #2 順序處理

- 需要先明確控管 `.github/workflows/deploy-pages.yml` 或 Pages 部署。
- deploy control 完成後，再依 stacked PR 順序處理 PR #1 -> PR #2。

優點：

- 保留 stacked PR merge 脈絡。
- 不需要重建整合 PR。
- PR #1 / PR #2 的 review breadcrumbs 最完整。

風險：

- 操作複雜。
- 若 deploy 控管失誤，PR #1 中間狀態可能上線。
- 需要 owner / maintainer 明確決策與權限。
- 停用 / 恢復 Pages deploy 可能影響其他部署流程。

適合：

- owner 能確實控管 deploy workflow。
- 團隊偏好保留 stacked PR 原始 merge 順序。

### 選項 D：暫停 release，改做後續 UI / readiness

- 不處理 merge。
- 先做同步生成進度 UI MVP、release note、rollback plan 或其他 release readiness 工作。

優點：

- 降低 deploy 前風險。
- 先補足使用者等待體感、release note 與 rollback plan。
- 避免在 release control 未明確前推進 main。

缺點：

- PR #1 / PR #2 繼續卡住。
- 需要另開後續任務或分支，避免污染 PR #2。

適合：

- owner 不急著 release。
- owner 希望先改善同步生成等待體驗或補齊 release readiness。

## 4. 建議

目前建議：

- 若 owner 尚未明確要 release：選 A。
- 若 owner 要推進 review，但不想讓 PR #1 單獨進 main：選 B。
- 不建議在未控管 Pages deploy 前選 C。
- 不建議現在 deploy。

固定判斷：

- PR #1 不應單獨 merge / deploy。
- PR #2 不應直接 merge 到 main。
- 建立整合 PR 不是 release approval，只是讓 main-facing review 有一個完整最終狀態。
- 即使選 B，整合 PR 仍應維持 Draft，直到 release risk review 完成。
