# 20260621 生成流程穩定化與 prompt 瘦身批｜規劃

## 一、背景

PR #1「命題提示詞品質增強」已完成 Run 1R A/B 驗收。

品質面結果：

- 新版人工可用率：100.0%。
- 新版直接可用率：100.0%。
- 新版 hard fail：0。
- 新版平均總分：20，高於舊版 18.67。
- 新版 `qualityMeta`：6/6 結構合規。

效能面結果：

- 新版平均生成時間較舊版增加 90.3%。
- 新版平均 prompt 長度較舊版增加 78.6%。
- 因此 PR #1 維持 Draft，不建議直接 merge / deploy。

目前判斷：

```text
品質目標已達成；效能目標成為導入門檻。
```

## 二、本批目標

在保留 PR #1 提示詞品質成果的前提下，降低新版 prompt 長度、生成時間與 JSON 失敗風險，使新版具備後續 merge / deploy 的工程條件。

本批不是重新設計命題品質規則，而是讓已驗收通過的品質版本變得更輕、更穩、更適合正式使用。

## 三、本批做

1. prompt 長度剖析，拆解各區塊對 promptLength 的貢獻。
2. prompt 區塊動態裁切，只載入當次科目、年級、題型需要的規則。
3. output / `qualityMeta` 瘦身，保留診斷價值但壓縮模型輸出成本。
4. few-shot 渲染瘦身，保留題感與迷思標註，壓縮全文與長解析。
5. `qualityMeta` contract 長度限制與必要性檢討。
6. JSON contract 去重與壓縮，保留硬規則，刪除重複說明。
7. 小樣本 A/B 效能複測，確認 promptLength、durationMs、success rate、qualityMeta 合規率。
8. 若瘦身後仍過慢，再設計單題重試、分批生成或進度 UI 方案。

## 四、本批不做

- 不新增新的 few-shot 題庫。
- 不擴自然科。
- 不處理題庫後台。
- 不做大型 UI 改版。
- 不改學校名稱、匯出格式等與效能批無關的功能。
- 不處理 npm audit vulnerability。
- 不動與本批無關的 stash 變更。
- 不把 Run 1 / Run 1R 的 raw outputs、blind key、完整 API 回應提交進 repo。

## 五、驗收指標

品質需維持：

- 新版人工可用率不低於舊版。
- 新版直接可用率不低於舊版。
- 新版 hard fail 不高於舊版。
- 新版 `qualityMeta` 結構合規。
- 學生版不外洩 `qualityMeta`、`distractorDesign`、`teacherExplanation` 或 `selfCheck`。

效能目標：

- 新版平均生成時間增幅由 +90.3% 降至 +30% 以內。
- 若 +30% 以內無法達成，需提出分批生成或進度 UI 方案，不得直接 deploy。
- 新版平均 promptLength 明顯下降。
- JSON parse failure 不增加。
- 生成成功率不低於 Run 1 的 12/12。

## 六、任務順序

### 任務 1：prompt 長度剖析

分析新版 prompt 各區塊長度：

- 基礎角色與任務規則。
- 題位資訊。
- 題型規範。
- few-shot 使用規則。
- few-shot 實例。
- 誘答迷思與品質規則。
- `qualityMeta` contract。
- JSON 穩定性規則。
- 最終輸出格式。

產出：

- 每個區塊的字元數與占比。
- 與舊版 prompt 的差異來源。
- 優先瘦身清單。

### 任務 2：prompt 動態裁切

依科目、年級、題型，只載入當次必要規則。

方向：

- 國語題不載入數學專屬題型規範。
- 數學題不載入國語向度、字詞短語、句式語法、段篇讀寫規則。
- 非閱讀題不載入完整閱讀測驗字數與題組規範。
- 非圖表題不載入圖表判讀與補圖提示規範。
- 非實驗探究題不載入實驗探究規範。

驗收：

- promptLength 下降。
- 對應題型仍保留必要規則。
- 測試確認不同 subject / questionType 會載入正確區塊。

#### 任務 2A / 2C 複測結果（已完成）

- 2A subject / questionType-aware prompt pruning 已完成。
- promptLength：10,437 → 6,919，下降 33.7%。
- durationMs：18,918 → 18,046，僅下降 4.6%。
- rawOutputLength 只下降 3.3%。
- qualityMetaLength 幾乎不降，甚至微增 0.5%。
- 12/12 生成成功。
- JSON parse failure：0。
- B1 qualityMeta：6/6 合規。
- B1 few-shot 科目隔離正常。

判斷：

```text
prompt pruning 有效降低輸入長度，但生成時間改善不足。
效能瓶頸可能主要在 output / qualityMeta 生成成本。
下一步應進入 output / qualityMeta 瘦身，不應繼續只砍 prompt input。
```

### 任務 3：output / qualityMeta 瘦身（下一步）

目標不是移除 `qualityMeta`，而是在保留診斷價值與結構合規的前提下，降低模型輸出長度與生成成本。

優先方向：

- 不要求模型重複輸出可由系統或題位補上的 metadata（如 subject、grade、unit、itemType、cognitiveLevel、difficulty 等），或改為可選並維持舊資料相容。
- 壓縮 `qualityMeta.teacherExplanation`、`qualityMeta.correctReason` 與 `distractorDesign` 各文字欄位長度。
- 保留 `qualityMeta.distractorDesign` 為物件，且每個錯誤選項仍需有 `misconceptionTag`。
- 保留 `qualityMeta.teacherExplanation` 必填。
- 本階段暫不改 `selfCheck` 結構，避免牽動 schema / validator / 前端顯示過大。

驗收：

- rawOutputLength 明顯下降。
- qualityMetaLength 明顯下降。
- JSON parse failure 維持 0。
- `qualityMeta` 合規率維持 100%。
- 學生版仍不外洩 `qualityMeta`、`distractorDesign`、`teacherExplanation` 或 `selfCheck`。

### 任務 4：few-shot 渲染瘦身

保留：

- `exampleId`。
- 科目、年級、題型。
- 對應能力或細目表格子。
- 這題好在哪。
- 誘答迷思摘要。
- 少量關鍵題幹或選項片段。

壓縮：

- 題目全文。
- 選項全文。
- 長解析。
- 過長 `reviewNote`。
- 重複的 `whyStudentsMayChooseIt` / `whyItIsWrong` 敘述。

可比較兩種模式：

| 模式 | 說明 |
|---|---|
| full example | 現行完整範例 |
| compact example | 題感摘要 + 誘答迷思摘要 |

驗收：

- compact mode 下品質不明顯下降。
- promptLength 明顯下降。
- few-shot 仍能傳遞題感與誘答設計判準。

### 任務 5：`qualityMeta` contract 瘦身

檢查哪些欄位必須強制生成，哪些可以縮短句長或移到教師模式。

優先檢討：

- `teacherExplanation` 是否需要長篇說明。
- `revisionNote` 是否可限制為短句。
- `whyStudentsMayChooseIt` 與 `whyItIsWrong` 是否可限制字數。
- `misconceptionDescription` 是否可避免與 `whyStudentsMayChooseIt` 重複。

不得破壞：

- `qualityMeta` 結構合規。
- `distractorDesign` 為物件且每個錯誤選項都有 `misconceptionTag`。
- `correctReason` 存在。
- 學生版投影隱藏內部命題資訊。

### 任務 6：JSON contract 去重與壓縮

保留：

- 必要 JSON schema 要求。
- 禁止 markdown fence。
- 選項格式與答案格式要求。
- `qualityMeta.distractorDesign` 必須為物件。
- 學生版不得外洩內部資訊。

壓縮：

- 重複的「只輸出 JSON」說明。
- 題型規範中與 schema 重疊的格式文字。
- 重複列出的品質檢核語句。

驗收：

- JSON parse failure 不增加。
- `G4_AB_CH_005` 類型的 JSON regression 不復發。
- `distractorDesign` 不退回陣列。

### 任務 7：小樣本 A/B 效能複測

沿用 Run 1 的 6 題題位包。

比較：

- promptLength。
- durationMs。
- success rate。
- JSON parse failure。
- `qualityMeta` 合規率。
- 學生版是否無內部資訊外洩。
- 題目品質是否維持。

建議分組：

| 組別 | 說明 |
|---|---|
| A | 舊版 main |
| B | PR #1 品質候選版 |
| C | 效能瘦身版 |

判讀：

- C 品質不得低於 B 太多。
- C 生成時間需明顯優於 B。
- 若 C 品質維持且效能接近 A，才具備導入條件。

## 七、prompt 長度剖析計畫

建立可重複執行的 prompt instrumentation：

1. 在 prompt builder 內標記每個文字區塊。
2. 產生 prompt 時同步輸出各區塊字元數。
3. 將區塊長度統計寫入 repo 外 A/B 工作目錄，不提交 raw prompt。
4. 比較 A / B / C 三組 promptLength 與 durationMs。

輸出欄位建議：

```json
{
  "slotId": "G4_AB_MA_001",
  "subject": "數學",
  "questionType": "選擇題",
  "totalPromptLength": 9753,
  "sections": {
    "baseRules": 1200,
    "slotContext": 900,
    "questionTypeGuidelines": 2200,
    "fewShotRules": 800,
    "fewShotExamples": 1800,
    "qualityMetaContract": 2100,
    "jsonContract": 753
  }
}
```

## 八、few-shot 瘦身方向

few-shot 不應完全移除，因 Run 1R 顯示品質候選已通過；但需要降低負擔。

優先策略：

1. 每次最多載入 1-2 題，而非固定最多 4 題。
2. 同科目、同題型優先；跨題型範例降權。
3. 使用 compact rendering：
   - 題幹縮成一句題型摘要。
   - 選項只保留正答與一個代表性誘答。
   - 迷思標籤保留完整。
   - 解析縮成 1-2 句。
4. 保留 `whyThisIsGood`，刪除過長示例文本。

風險：

- few-shot 過度縮短可能使題感下降。
- 需要用小樣本 A/B 驗證品質是否維持。

## 九、`qualityMeta` contract 瘦身方向

可先保留 schema，不改資料結構，只限制生成文字長度。

建議限制：

- `correctReason`：1-2 句。
- `teacherExplanation`：2-4 句。
- 每個 `distractorDesign` 欄位各 1 句。
- `revisionNote` 可改為簡短命令句，或允許空字串。

暫不建議：

- 移除 `misconceptionTag`。
- 移除 `distractorDesign`。
- 把 `qualityMeta` 改成 optional。

原因：Run 1R 的品質提升與 `qualityMeta`、誘答診斷資料相關，不能在第一輪效能優化中直接拆掉品質核心。

## 十、小樣本 A/B 效能複測計畫

使用與 Run 1 相同題位包，避免題位差異干擾。

複測步驟：

1. 產生 A / B / C 三組 promptLength 與生成時間。
2. 確認 C 組 JSON parse 成功。
3. 檢查 C 組 `qualityMeta` 合規。
4. 建立去識別盲評檔。
5. 若 C 組品質疑似下降，再進行 Gemini 盲評；若品質明顯維持，可先做人工抽查。

指標：

| 指標 | 目標 |
|---|---|
| C 平均 promptLength | 明顯低於 B |
| C 平均 durationMs | 較 B 顯著下降 |
| C 成功率 | 不低於 B |
| C JSON parse failure | 不高於 B |
| C qualityMeta 合規率 | 100% |
| C 學生版外洩 | 0 |

## 十一、導入判斷

若瘦身版 C 同時符合：

- 品質不低於 B 或只小幅下降但仍不低於 A。
- `qualityMeta` 仍 100% 合規。
- 平均生成時間增幅降至 +30% 以內。
- JSON failure 不增加。

則可考慮將 PR #1 或效能批轉入 merge 前最後審查。

若品質維持但生成時間仍大幅增加：

```text
不直接 deploy；進一步規劃分批生成或進度 UI。
```

若生成時間下降但品質明顯退步：

```text
回調 few-shot 或 qualityMeta contract 瘦身幅度，重新複測。
```
