# 生成進度 UI 規格

- 日期：2026/6/21
- 分支：`codex/generation-stabilization-20260621`
- 任務：5B
- 性質：規格文件。此文件不代表已實作，不改功能程式、不改 prompt、不重跑生成。

## 1. 背景與問題定義

B4 已通過 structure gate：標準整卷 12/12 成功，國語 6/6、數學 6/6，且 JSON parse、validation、answer contract、distractorDesign key contract、qualityMeta、student leakage 均為 0 failure。

但 deploy gate 仍未通過：

- B4 標準整卷 totalDuration 為 92.1s。
- B4 相對 main totalDuration 增加 26.4%。
- B4 raw output 相對 main 增加 181.6%。
- 目前 deploy blocker 不是題目結構，而是等待體感、latency 與 output / token cost。

進度 UI 的目標是降低等待焦慮、提升可預期性，讓老師知道系統正在工作、目前大約處在哪個階段、如果失敗下一步可以怎麼做。進度 UI 不能降低實際生成時間，也不能降低 token 成本；它是產品體感與錯誤承接方案，不是效能優化本身。

## 2. 使用者情境

### 情境 1：老師產生 6 題小型練習卷

- 使用者想完成什麼：快速產生一份課堂練習或補救教學小卷。
- 使用者可能焦慮點：等待超過 30 秒時，會懷疑是否卡住或網路中斷。
- UI 應提供哪些資訊：顯示「正在生成練習題」、「已送出 6 題需求」、「通常需要 30-60 秒」。
- 不應顯示哪些技術細節：不顯示 API、raw prompt、model response、token、stack trace。

### 情境 2：老師產生 10-12 題標準整卷

- 使用者想完成什麼：產生可進入修題定稿流程的完整試卷草稿。
- 使用者可能焦慮點：等待接近 90 秒時，會不確定是否該重按或離開頁面。
- UI 應提供哪些資訊：顯示目前是標準整卷、題數較多、可能需要 1-2 分鐘；提供階段狀態與等待提示。
- 不應顯示哪些技術細節：不顯示內部 `qualityMeta`、原始 JSON、錯誤堆疊、API timeout 細節。

### 情境 3：生成中途某一題失敗

- 使用者想完成什麼：知道是哪一題或哪一批出問題，並盡量保留已完成成果。
- 使用者可能焦慮點：擔心整份卷都要重來，或不知道錯誤是否可修復。
- UI 應提供哪些資訊：若未來支援分批，顯示失敗題位、題型、錯誤摘要與「重試失敗題」；同步版則顯示整卷失敗原因與重試建議。
- 不應顯示哪些技術細節：不顯示 validator 完整錯誤陣列、raw output、prompt。

### 情境 4：生成時間超過預期，但尚未失敗

- 使用者想完成什麼：確認系統仍在處理，並知道是否需要繼續等待。
- 使用者可能焦慮點：90 秒以上容易被解讀為當機。
- UI 應提供哪些資訊：30 / 60 / 90 秒階段提示、已等待時間、保持頁面開啟的提醒。
- 不應顯示哪些技術細節：不顯示「模型延遲」「token output 過大」等非教師可操作資訊。

## 3. 進度狀態設計

| 狀態 | 說明 | 顯示文字 | 使用者可否操作 | 已完成題數 | 預估剩餘時間 | 重試按鈕 | 保留已完成結果 |
|---|---|---|---|---|---|---|---|
| idle | 尚未開始 | 尚未開始生成 | 可編輯命題條件 | 不顯示 | 不顯示 | 不顯示 | 無 |
| preparing | 整理命題條件 | 正在整理命題條件與題位 | 不建議操作 | 不顯示 | 不顯示 | 不顯示 | 無 |
| generating | 正在生成 | 正在產生試題草稿 | 可查看等待提示，不可重複送出 | 同步版不顯示；分批版顯示 | optional | 不顯示 | 分批版保留 |
| validating | 正在檢查題目格式 | 正在檢查題目格式與答案選項 | 不建議操作 | 可顯示已生成題數 | 不顯示 | 不顯示 | 保留 |
| partial-complete | 部分題目完成 | 已完成部分題目，仍在處理剩餘題目 | 可預覽已完成題目 | 顯示 | optional | 失敗題可重試 | 保留 |
| completed | 全部完成 | 題目草稿已完成 | 可進入修題 | 顯示全部 | 不顯示 | 不顯示 | 保留 |
| failed | 生成失敗 | 生成未完成，請重試或調整條件 | 可重試、返回修改 | 若有 partial 則顯示 | 不顯示 | 顯示 | 分批版保留成功題 |
| retrying | 重新嘗試 | 正在重新嘗試失敗題目 | 暫停其他送出 | 顯示 | optional | 不顯示 | 保留 |
| cancelled | 使用者取消 | 已取消生成（目前未實作） | 可返回或重新開始 | 若有 partial 則顯示 | 不顯示 | 可重新開始 | 未實作 |

`cancelled` 目前標記為未實作。同步生成時若 request 已送出，前端即使離開畫面也不一定能取消後端請求，因此 MVP 不提供真正取消，只提供返回提醒。

## 4. 進度呈現方式

### 4.1 整體進度列

同步生成 MVP：

- 顯示階段，不顯示精準百分比。
- 顯示「已送出 12 題命題需求」。
- 顯示已等待秒數。
- 若無 item-level progress，不顯示「已完成 4 / 12 題」，避免誤導。

分批生成版本：

- 顯示 `已完成 4 / 12 題`。
- 顯示百分比，例如 `33%`。
- 顯示目前批次，例如 `正在生成第 5-6 題`。
- 若題組不可拆批，顯示為同一組處理。

### 4.2 題目狀態清單

每題狀態：

- `waiting`：尚未開始。
- `generating`：正在生成。
- `validating`：正在檢查格式。
- `done`：完成。
- `failed`：失敗。

每題建議顯示：

- 題號或題位。
- 科目。
- 題型。
- 難度。
- 狀態。
- 失敗時顯示錯誤摘要。

同步 MVP 不一定顯示每題狀態清單；可先顯示整體 loading panel。分批 POC 再加入每題狀態。

### 4.3 等待提示文字

可直接用於 UI 的繁體中文文案：

1. `正在產生試題草稿，題數較多時可能需要 1-2 分鐘。`
2. `系統正在整理題位、命題條件與教材內容，請先保持此頁面開啟。`
3. `AI 正在產生題目與解析，完成後會自動進入修題畫面。`
4. `目前仍在處理中，若等待超過平常時間，系統會提示您是否重試。`
5. `正在檢查題目格式、答案與選項，請稍候。`

文案風格應偏教育行政工具：清楚、穩定、可預期。不使用「點亮夢想」「展現無限可能」等宣傳語。

### 4.4 錯誤提示文字

單題失敗：

1. `第 {itemNumber} 題生成未完成。您可以重試此題，或先保留其他已完成題目。`
2. `{questionType} 題目未通過格式檢查，建議重試此題或調整命題條件。`

整卷失敗：

1. `本次試卷生成未完成。請稍後重試，或減少題數後再產生。`
2. `系統未能完成整份試卷草稿。已保留目前設定，您可以重新送出生成。`

格式檢查失敗：

1. `題目已產生，但部分欄位格式不符合要求。請重試，或改用較少題數產生。`
2. `系統檢查到答案、選項或解析格式不完整，建議重新生成該題。`

網路或 API 逾時：

1. `連線等待時間較長，系統未收到完整回應。請稍後重試。`
2. `AI 服務回應逾時。您的命題條件已保留，可以重新產生。`

## 5. 同步生成下的 UI 規格

目前若仍是同步生成，request 送出後前端應：

1. 鎖定生成按鈕，避免重複送出。
2. 顯示 loading panel。
3. 顯示目前階段：`preparing -> generating -> validating`。
4. 顯示已等待時間。
5. 成功後一次呈現結果。
6. 失敗後顯示可理解的錯誤訊息。

若後端無法回傳 item-level progress，前端只能顯示階段型提示，不應顯示假的題目完成數。可使用「非精準」提示，例如：

- `正在生成題目`
- `正在整理回傳內容`
- `正在檢查題目格式`

超過不同等待時間時：

| 等待時間 | 提示 |
|---:|---|
| 30 秒 | `題目仍在產生中，標準整卷可能需要較長時間。` |
| 60 秒 | `目前仍在處理，請保持頁面開啟；完成後會自動顯示結果。` |
| 90 秒 | `等待時間較長。若最後失敗，您可以保留設定後重試。` |

使用者是否可以取消：

- MVP 不做真正取消。
- UI 可提供「返回修改」提示，但要說明離開頁面可能中斷本次結果。
- 不提供會誤導的「取消後端任務」按鈕。

同步生成限制：

- 無法顯示真實 item-level progress。
- 單次請求失敗時，通常需要整份重跑。
- 無法支援 partial result。
- 使用者重新整理後難以恢復。

## 6. 分批生成下的 UI 規格

未來分批生成時，建議每批 2 題或 3 題。

UI 呈現：

- 整體進度：`已完成 4 / 12 題`。
- 批次狀態：`正在生成第 5-6 題`。
- 每題狀態：waiting / generating / validating / done / failed。
- 已完成題目可先顯示在草稿列表中。
- 未完成題目以 skeleton 或題位卡占位。

某批失敗時：

- 標示該批或該題 failed。
- 保留已完成批次。
- 提供「重試失敗題」或「重試此批」。
- 不要求整卷重跑。

題目順序：

- 依原題位排序呈現。
- 即使後完成，也放回原題號位置。
- 題組不可拆批，避免 stimulus / groupId / subScores 斷裂。

使用者是否能先預覽部分題目：

- 可以，但需標示「草稿生成中」。
- 未完成前不應允許最終輸出 Word / Excel。

體感改善：

- 老師會看到題目逐步完成。
- 失敗成本降低。
- 即使總時間不一定下降，等待感會更可控。

## 7. 非同步生成下的 UI 規格

非同步 job 模式下，使用者送出整卷需求後，後端建立 job 並回傳 `jobId`。

UI 表現：

| job 狀態 | UI 表現 |
|---|---|
| pending | 已建立生成任務，等待開始 |
| running | 正在生成題目，顯示已完成題數 |
| partial | 部分題目已完成，可預覽草稿 |
| completed | 全部完成，可進入修題 |
| failed | 任務失敗，可重試或返回修改 |

重新整理頁面後：

- 前端從 localStorage 或 project state 取回 `jobId`。
- 呼叫 job status API。
- 恢復 partial / completed / failed 狀態。

是否需要稍後查看：

- 若任務可能超過 2 分鐘，應提供「您可以稍後回來查看」提示。
- 完成提示可先用畫面內通知，不必第一版做 email 或系統通知。

失敗 job retry：

- 優先 retry failed batch。
- 若 job metadata 不完整，才 retry entire job。
- 顯示 retry 次數與最後錯誤摘要。

完成結果保存：

- 需暫存 items、validation result、建立時間、過期時間。
- 需定義結果保存期限與清理機制。

## 8. 資料需求

| 欄位 | 型別 | 必填 | 用途 | 備註 |
|---|---|---|---|---|
| jobId | string | 是 | 查詢生成任務 | 同步 MVP 可無 |
| status | string | 是 | 顯示目前狀態 | idle / preparing / generating / validating / partial / completed / failed |
| totalItems | number | 是 | 計算進度 | 同步 MVP 可使用題位數 |
| completedItems | number | 是 | 顯示已完成題數 | 同步 MVP 可不顯示 |
| failedItems | number | 是 | 顯示失敗題數 | 同步 MVP 失敗時可為總失敗 |
| currentItemIndex | number | 否 | 顯示目前題號 | 分批或非同步較需要 |
| currentItemSubject | string | 否 | 顯示目前科目 | 混合題位或分科生成時使用 |
| currentItemType | string | 否 | 顯示目前題型 | 題目狀態清單使用 |
| itemStatuses | array | 否 | 每題狀態 | 分批 / 非同步需要 |
| startedAt | string | 是 | 計算已等待時間 | ISO datetime |
| updatedAt | string | 是 | 判斷狀態是否停滯 | ISO datetime |
| estimatedRemainingSeconds | number | 否 | 顯示預估剩餘時間 | optional；不準確時不要顯示 |
| errors | array | 否 | 顯示錯誤摘要 | 不含 raw output / stack trace |
| partialItems | array | 否 | 顯示已完成草稿 | 分批 / 非同步需要 |

## 9. 最小可行版本 MVP

若進入 5B-impl，MVP 建議包含：

- 同步生成 loading panel。
- 顯示階段型提示。
- 顯示已送出命題需求。
- 顯示已等待時間。
- 超過 30 / 60 / 90 秒顯示不同提示。
- 完成後一次呈現結果。
- 失敗時提供明確錯誤文案。

MVP 不做：

- 不做真正 item-level progress。
- 不做取消。
- 不做 partial result。
- 不做 job queue。
- 不做 Durable Object / KV。

先做 MVP 的理由：

- 目前 B4 結構已穩，最短路徑是改善同步等待體感。
- 完整非同步會大幅擴張架構範圍。
- 分批與非同步需要更多資料流設計，應先有 UI 狀態語言。
- MVP 可快速降低「看起來像當機」的問題。

## 10. 驗收標準

未來實作時至少需滿足：

- 生成開始後 UI 不會像當機。
- 使用者知道系統正在生成。
- 超過 30 秒有提示。
- 超過 60 秒有提示。
- 超過 90 秒有提示。
- 成功後正常呈現試卷。
- 失敗後有可理解錯誤訊息。
- 不顯示 API key、raw prompt、raw output。
- 不向學生版外洩 `qualityMeta`。
- 手機與桌面都能閱讀。
- 生成按鈕在處理中不會重複送出。
- 錯誤文案提供下一步行動。

## 11. 不做事項

- 不實作真正非同步 job queue。
- 不實作 Durable Object / KV 儲存。
- 不實作取消生成。
- 不實作 partial result。
- 不改 prompt。
- 不改 schema。
- 不重跑生成。
- 不處理 npm audit。
- 不 merge / deploy。

## 12. 後續任務建議

### 5B-impl：同步生成進度 UI MVP 實作

- 目標：在現有同步生成流程中加入 loading panel、階段提示與 30 / 60 / 90 秒等待文案。
- 修改範圍：前端 UI、state、錯誤顯示。
- 驗收標準：長等待時畫面可理解，成功 / 失敗均有明確狀態。
- 風險：只改善體感，不降低成本與實際時間。

### 5C：output / qualityMeta budget diagnostics

- 目標：建立 rawOutputLength、qualityMetaLength、teacherExplanationLength、distractorDesignLength 等 budget 與 diagnostics。
- 修改範圍：診斷規格、生成後統計、repo 外報表或 debug panel。
- 驗收標準：能找出哪類題目最胖，能判斷整卷是否超 budget。
- 風險：budget 設太硬可能傷害品質。

### 5D：分批生成 POC

- 目標：驗證每 2-3 題一批的生成流程、partial results 與 failed item retry。
- 修改範圍：POC 腳本或 feature branch，batch parse / normalize / validate，整卷合併檢核。
- 驗收標準：標準 12 題可分批成功，題組不被拆散，失敗批次可重試。
- 風險：API 呼叫次數增加，總時間可能不降反升。

### 5E：非同步 job queue 技術評估

- 目標：評估 Durable Object / KV / Queue / Workflow 等 job 架構。
- 修改範圍：技術評估文件、job 狀態模型、資料保存與恢復策略。
- 驗收標準：有清楚選型、狀態流、retry、resume、清理與成本評估。
- 風險：改動面大，容易把本批擴成平台架構工程。

## 13. 總結

B4 結構穩，品質方向成立，但不宜 deploy。5B 的任務是先把老師等待時看到的狀態、提示與錯誤承接定清楚。接著再做 5C output / qualityMeta budget diagnostics，處理目前最硬的 deploy blocker：raw output 增加 181.6%。
