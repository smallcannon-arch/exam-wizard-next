# 生成等待 UX 與分批 / 非同步生成方案規劃

- 日期：2026/6/21
- 分支：`codex/generation-stabilization-20260621`
- 任務：5A
- 性質：規劃文件。此文件不代表已實作，不改功能程式、不改 prompt、不重跑生成。

## 1. 背景與問題定義

B4 已通過標準整卷 structure gate：12 題標準整卷全數成功，國語 6/6、數學 6/6，且 JSON parse、validation、answer contract、distractorDesign key contract、qualityMeta、student leakage 均為 0 failure。

但 deploy gate 尚未通過：

| 指標 | main | B4 | 判斷 |
|---|---:|---:|---|
| totalDuration | 72.9s | 92.1s | B4 增加 26.4%，低於 +30% duration gate，但接近風險邊界 |
| raw output | baseline | +181.6% | 超過 +100% output / cost gate，仍是 deploy blocker |

因此目前主要問題已從「命題品質 / 結構穩定性」轉為：

- 等待體感：老師是否願意等 90 秒以上。
- token 成本：`qualityMeta` 與 few-shot 讓 raw output 明顯膨脹。
- 生成流程韌性：長時間單次呼叫若失敗，整卷重跑成本高。
- 可恢復性：使用者切頁或重新整理後，是否能看見已完成進度。

目前不建議 merge / deploy。下一階段應先規劃生成等待 UX、分批生成與 output budget 控制，再決定是否需要完整非同步架構。

## 2. 方案 A：進度 UI

### 適用情境

適合 B4 目前狀態：結構穩定、仍需等待，但不一定要立即改生成架構。若短期內仍採整卷同步生成，進度 UI 是最低風險的體感改善。

### 預期改動範圍

- 前端：新增生成狀態區塊、進度提示、等待文案、取消 / 重試提示。
- 前端 state：記錄目前步驟、已等待秒數、目前生成階段。
- 後端：若仍是單次同步 API，後端可不改；若要真實 item-level progress，後端需配合分批或 job 狀態。

### 前端需要顯示的狀態

- `準備命題資料`
- `連線 AI`
- `生成題目中`
- `整理題目格式`
- `檢核答案與選項`
- `完成`
- `失敗，可重試`

若尚未分批，進度應標示為階段式或等待式，不應偽裝成真實百分比。建議顯示「已等待 N 秒」與「此步通常需要 1-2 分鐘」。

### 後端是否需改 API

第一階段不一定需要。只做等待體感改善時，可維持現有 API。若要顯示每題完成狀態，必須改為分批生成或非同步 job。

### 是否能降低實際 totalDuration

不能。進度 UI 不會降低模型生成時間、token 使用量或 raw output。

### 是否能改善等待體感

可以。它能降低使用者誤判系統卡死的機率，也能讓長等待變得可預期。

### 主要風險

- 如果進度顯示不真實，反而降低信任。
- 若沒有取消 / 重試策略，長等待失敗仍會挫折。
- 若只做 UI 不處理 output / cost，deploy blocker 仍在。

### 實作成本估計

低到中。

### 建議是否採用

建議採用，作為第一階段。它不能解決成本，但能先承接使用者等待體感，且不會大幅牽動架構。

## 3. 方案 B：分批生成

### 流程

分批生成可將整卷題位拆成多批，例如每 2 題或每 3 題一批：

1. 前端依題位建立 batch list。
2. 逐批呼叫生成 API。
3. 每批回傳後立即 parse、normalize、validate。
4. 成功題先顯示於修題區或暫存區。
5. 失敗 batch 可單獨 retry，不需整卷重跑。
6. 全部 batch 完成後再做整卷級統計檢核。

### batch size

| batch size | 優點 | 風險 |
|---|---|---|
| 每 1 題 | 最容易 retry，單次 timeout 風險最低 | API 次數最多，總時間可能最長，整卷一致性較差 |
| 每 2 題 | 兼顧 retry 與上下文，適合作 POC | 呼叫次數仍增加 |
| 每 3 題 | 呼叫次數較少，仍可局部恢復 | 單批失敗重跑成本較高 |
| 依科目分批 | 符合目前 subject-specific 流程 | 國語 6 題仍可能單批過長 |

建議 POC 先試每 2-3 題一批。不要一開始做逐題生成，避免 API 次數與整卷一致性問題過大。

### 前端如何顯示已完成題目

- 每批完成後更新題位狀態：`pending / running / completed / failed`。
- 已完成題目可先進入「草稿待檢核」狀態。
- 顯示已完成題數，例如 `已完成 4 / 12 題`。
- 失敗題位提供「只重試失敗題」。

### 失敗題目如何 retry

- 保留原題位與 objective / cognitiveLevel / score。
- 只重跑 failed batch。
- retry 後仍需 normalize / validate。
- 若 retry 多次失敗，允許老師手動改題或降級為單題重出。

### 是否能降低單次 timeout 風險

能。單次 prompt 與 output 變短，timeout 與 JSON 失敗機率通常會下降。

### 是否能改善使用者體感

能。使用者可以看到題目逐批完成，而不是等待整卷一次回來。

### 是否會增加 API 呼叫次數

會。這是主要成本之一。batch 越小，呼叫次數越多。

### 是否會增加總時間

可能。若批次串行執行，總時間可能增加；若可有限並行，總時間可控制，但會增加後端與 rate limit 風險。

### 對 schema / normalize / validate 流程的影響

- 單批仍可沿用現有 parse -> normalize -> validate。
- 需新增整卷合併後檢核：題數、總分、題位覆蓋、目標覆蓋、題組完整性。
- 若題組跨 batch，風險高；建議題組不可拆批。

### 對 qualityMeta 的影響

每題仍保留 `qualityMeta`。但 batch prompt 變短後，模型可能更穩定輸出完整 `qualityMeta`。若 output 成本仍高，需搭配 budget diagnostics。

### 實作成本估計

中。

### 建議是否採用

建議採用，但先做 proof of concept，不直接改正式流程。POC 目標是量測 batch size 對成功率、總時間、raw output 與使用者體感的影響。

## 4. 方案 C：非同步生成

### 流程

非同步生成代表使用者送出整卷需求後，後端建立 job：

1. 前端送出生成請求。
2. 後端回傳 `jobId`。
3. job 進入 `pending`。
4. worker 執行生成，狀態變為 `running`。
5. 可逐批產生 `partial` 結果。
6. 全部完成後變為 `completed`。
7. 失敗時變為 `failed`，並保留錯誤與可 retry 狀態。

### job 狀態

至少需要：

- `pending`
- `running`
- `partial`
- `completed`
- `failed`

可加上：

- `cancelled`
- `expired`
- `retrying`

### 結果如何暫存

需要儲存：

- job metadata：科目、年級、題位、建立時間、狀態。
- partial items：已成功題目。
- errors：失敗 batch、validation errors。
- final result：完整 items 與檢核結果。

### 失敗如何 retry

retry 應支援：

- retry failed batch。
- retry entire job。
- 保留成功題，只重跑失敗題。
- retry 次數上限與錯誤紀錄。

### 使用者重新整理頁面後如何恢復

前端需要保存 `jobId`，例如 localStorage 或 project state。重新整理後用 `jobId` 查詢狀態，恢復 partial result 或 completed result。

### 是否需要儲存

需要。Cloudflare Workers 架構下可考慮：

- KV：適合 job 狀態與結果快取，但一致性與更新頻率需注意。
- Durable Object：適合 job 狀態機、進度更新、單 job 協調。
- R2：適合較大的 raw output 或 artifact，但不適合頻繁小狀態更新。
- D1：適合需要查詢與歷史紀錄，但導入成本較高。

若要做完整非同步，Durable Object 或 Worker + KV 組合較合理。若需要長時間任務與可靠重試，還需評估 Cloudflare Queues / Workflows。

### 對 Cloudflare Workers 架構的影響

- 需要新增 job API：create / read / retry / cancel。
- 需要狀態儲存 binding。
- 需要處理 timeout、重試、冪等性。
- 需要避免把 API key 或原始 prompt 暴露到前端。
- 需要清理過期 job。

### 對 GitHub Pages 前端的影響

- 前端需輪詢 job 狀態或使用可用的訂閱機制。
- 需要 job 恢復 UI。
- 需要 partial result 顯示與錯誤重試 UI。
- 需要處理使用者離開頁面後再回來的狀態。

### 實作成本估計

高。

### 建議是否採用

暫不作為下一步直接實作。先做進度 UI 與分批 POC，確認同步分批是否足以解決等待與失敗恢復。若仍無法接受，再進非同步 job queue。

## 5. Output / cost 控制策略

B4 raw output 相對 main 增加 181.6%，主因是 `qualityMeta`、誘答診斷與更完整的結構化說明。這些資料對品質有價值，不應直接移除，但必須建立 budget。

### 可行方向

1. 進一步壓縮 `qualityMeta`
   - 限制 `correctReason` 為 1-2 句。
   - 限制 `teacherExplanation` 為 2-4 句。
   - 限制每個 `distractorDesign` 子欄位為 1 句。
   - 限制 `revisionNote` 為短句。

2. tiered detail
   - `compact`：預設模式，保留必要診斷欄位，但字數短。
   - `full`：教師審題或 debug mode 才生成完整說明。
   - `student`：不含內部命題資訊。

3. 教師版保留完整資訊
   - 不把 `qualityMeta` 洩漏到學生版。
   - 教師審題可看到完整或展開式診斷。

4. rawOutputLength budget
   - 為整卷與單題建立預算。
   - 例如單題 raw output 超過指定長度時列 warning。

5. qualityMetaLength budget
   - 記錄每題 `qualityMetaLength`。
   - 若超過預算，列入 diagnostics，不一定當場 fail。

6. 生成後 diagnostics
   - 記錄 promptLength、rawOutputLength、studentItemLength、qualityMetaLength。
   - 以 repo 外報表或前端 debug panel 顯示。

### 不建議

- 不建議直接移除 `qualityMeta`。
- 不建議取消 `distractorDesign`。
- 不建議把誘答迷思標籤改回純文字解析。

理由：`qualityMeta` 是 PR #1 品質提升的重要成果，應先壓縮與分層，而不是拆掉。

## 6. 方案比較表

| 方案 | 降低等待體感 | 降低實際生成時間 | 降低 token 成本 | 降低 timeout 風險 | 實作成本 | 風險 | 建議 |
|---|---|---|---|---|---|---|---|
| 進度 UI | 高 | 無 | 無 | 無 | 低到中 | 進度若不真實會降低信任 | 先做 |
| 分批生成 | 高 | 不一定，串行可能增加 | 不一定 | 高 | 中 | API 次數增加、整卷一致性需補檢核 | 做 POC |
| 非同步生成 | 高 | 不一定 | 不一定 | 高 | 高 | 狀態儲存、job queue、恢復與重試複雜 | 暫緩，待 POC 後評估 |
| output / cost budget | 中 | 可能降低 | 高 | 中 | 中 | 壓太短可能傷害題目品質 | 與進度 UI 並行規劃 |

## 7. 建議路線

### 階段一

- 保留 B4。
- 不 deploy。
- 先做進度 UI + item-level progress 規格。
- 同時建立 output / qualityMeta budget diagnostics。

理由：B4 結構已穩，第一階段應先改善等待可理解性，並把 output 膨脹量化，避免盲目砍品質資料。

### 階段二

- 做分批生成 proof of concept。
- 優先嘗試每 2-3 題一批。
- 支援 partial results 與 failed item retry。
- 題組不可拆批。

理由：分批是降低單次 timeout 風險與支援局部 retry 的關鍵，但它會增加 API 呼叫與整卷合併檢核複雜度，需要 POC 量測。

### 階段三

- 若整卷仍超過可接受等待時間，再規劃非同步 job queue。
- 評估 Durable Object / KV / Queue / Workflow 的搭配。

理由：完整非同步會一次碰到 job queue、狀態儲存、重試、恢復、前端輪詢與結果暫存。現在直接做會讓改動面過大，且還不知道同步分批是否已足夠。

## 8. 下一批任務拆解

### 5B：生成進度 UI 規格

目標：

- 定義同步生成與分批生成兩種情境下的使用者等待畫面。
- 明確列出狀態、文案、錯誤、重試與取消行為。

不做事項：

- 不改後端生成流程。
- 不做完整非同步 job。

修改範圍：

- 規格文件。
- 可能補前端狀態設計草圖，但不直接實作功能。

驗收標準：

- 能描述使用者從按下生成到完成 / 失敗的完整狀態。
- 不使用假百分比。
- 有等待時間提示與可重試策略。

風險：

- 若只做 UI，成本與實際時間不會下降。

### 5C：output / qualityMeta budget diagnostics

目標：

- 建立 promptLength、rawOutputLength、studentItemLength、qualityMetaLength 的統計與預算。
- 找出 output 膨脹主要來源。

不做事項：

- 不移除 `qualityMeta`。
- 不直接改品質 schema。

修改範圍：

- diagnostics 規格。
- 可在後續實作 repo 外報表或前端 debug panel。

驗收標準：

- 每次生成可量測長度。
- 超出 budget 時可被記錄。
- 能區分學生版長度與教師內部資訊長度。

風險：

- 若 budget 設太硬，可能誤擋品質較好的題目。

### 5D：分批生成 POC

目標：

- 驗證每 2-3 題一批是否能改善等待體感、降低 timeout 風險，並支援 partial results / failed batch retry。

不做事項：

- 不做完整非同步 job queue。
- 不改正式 deploy 流程。

修改範圍：

- POC 腳本或 feature branch 實作。
- 生成流程切批、合併檢核與局部 retry。

驗收標準：

- 12 題標準整卷可分批成功。
- batch-level parse / normalize / validate 正常。
- 合併後整卷檢核正常。
- 可量測每批 duration 與 output。
- 題組不被拆散。

風險：

- API 呼叫次數增加。
- 總時間可能不降反升。
- 需要處理跨批一致性與整卷統計。

### 5E：非同步 job queue 技術評估

目標：

- 評估若分批仍不足，如何導入 job queue 與狀態儲存。

不做事項：

- 不直接建完整 production queue。
- 不 deploy。

修改範圍：

- 技術評估文件。
- Cloudflare Workers / Durable Object / KV / Queue / Workflow 選型。
- 前端輪詢與恢復策略。

驗收標準：

- 有明確架構選項與取捨。
- 有 job 狀態模型。
- 有資料保存、清理、retry、resume 策略。
- 能評估成本與改動面。

風險：

- 改動大，容易把 prompt 品質批擴成平台架構批。
- 需要處理資料一致性與長任務可靠性。

## 9. 總結

B4 已通過標準整卷 structure gate，修復 4C 的國語 answer / distractorDesign contract 阻擋。B4 相對 main 的 totalDuration 增幅降至 +26.4%，但 raw output 仍增加 +181.6%。因此目前不建議 merge / deploy；下一階段應轉向生成等待 UX、分批生成與 output budget 控制。
